Architecture Decisions & Constraints

Stream mode cascade: WebRTC → JPEG WS → MJPEG
WebRTC delivers H.264 with hardware encoding and lowest latency. JPEG WS avoids HTTP connection overhead vs MJPEG. MJPEG is the universal fallback that works on any browser without extra negotiation. The client manages three distinct lifecycles and falls back automatically on any failure.
Single in-process state singleton (no DB)
Avoids external dependencies for a single-user local tool. threading.Lock is sufficient. Implication: server restart resets all state except persisted config. Not scalable beyond one session.
Capture pauses when active_streams == 0
No point capturing frames when no client is watching. active_streams counter is incremented on stream connect, decremented on disconnect. At 0, dxcam camera is stopped and clear_latest_frame() is called. First frame after reconnect may be slightly delayed.
config_version integer for cache-busting
Both the capture loop and the WebSocket input handler need to react to settings changes (monitor switch, scale change). A version integer that increments on every update_config() lets all hot-path loops detect changes on the next tick without an event queue or pub/sub system.
Token as ?token= query param (not Authorization header)
WebSocket connections and <img src=> MJPEG cannot set custom headers — query param is the only portable option. Token is visible in server logs and browser history, but acceptable for local LAN use. Mitigated by token being random 24-byte urlsafe string.
Coordinate normalization [0,1] over the wire
Decouples iPad viewport size from Windows monitor resolution. Server recomputes absolute pixel coordinates from current monitor dimensions. WebSocket handler refreshes these on config_version change, so monitor switches are transparent to the client.
dxcam preferred over mss — output_idx = monitor_index - 1
dxcam uses DXGI (DirectX desktop duplication) — significantly lower latency and CPU than GDI-based mss. Critical gotcha: dxcam indexes monitors 0-based (output_idx) while mss indexes 1-based (monitor_index). The mapping is output_idx = monitor_index - 1.
Auto-quality runs entirely on the client
Client observes encode_ms and size_bytes from GET /stats and decides when to throttle via PATCH /settings. Avoids server-side complexity at the cost of 1s polling lag. Caveat: two iPads connected could fight over quality settings.

Hard Constraints

Windows-only
dxcam = DXGI API (Windows). pyautogui input injection = Win32. mss works cross-platform but capture + input would need replacement on macOS/Linux.
Single session token
SESSION_TOKEN is one value shared by all clients. Multiple iPads share the same token and can control the same cursor. No per-client isolation.
No HTTPS by default
Wake Lock and Clipboard API require secure context. On LAN via IP, HTTPS needs a self-signed cert. navigator.clipboard.readText will silently fail on http://.
pyautogui.PAUSE = 0
Eliminates the 0.1s default delay between all pyautogui calls. Necessary for responsive input, but removes all safety pauses globally.
WebRTC ScreenVideoTrack uses own mss
Independent of the shared capture loop. Runs at display framerate, not config fps. Can create competing capture load alongside the main loop.
config.local.json: single-process only
Atomic tmp-rename write handles single-process races. Would break under multi-worker uvicorn (not used currently, but worth noting if scaling).
